# Lobe Chat v2 迁移紧急回滚计划

## 回滚前提条件
仅在以下情况下执行回滚：
1. v2 版本出现严重功能故障且无法快速修复
2. 认证系统完全无法工作
3. 数据丢失或损坏

## 回滚步骤

### 阶段1：停止 v2 服务
```bash
# 将 Lobe Chat 副本数设为0
kubectl scale deployment/lobe --replicas=0 -n lobe-chat

# 将 Casdoor 副本数设为0（如果需要）
kubectl scale deployment/casdoor --replicas=0 -n lobe-chat
```

### 阶段2：恢复数据库
**重要**：数据库已从 PostgreSQL 16 升级到 17，无法直接降级。回滚方案：
1. 使用备份文件恢复数据到新的 PostgreSQL 16 实例
2. 或继续使用 PostgreSQL 17，但恢复 v1 应用配置

**备份文件位置**：
- 完整数据库备份：`lobechat_v1_backup_20260314_182103.dump`
- 用户表备份：`users_backup_20260314_182200.sql`
- Secrets 备份：`secrets_backup_20260314_182304.yaml`

### 阶段3：恢复 v1 配置
1. 恢复 deployment 配置：
   ```bash
   # 应用备份的 v1 配置
   kubectl apply -f deploy-lobe.yaml.backup_20260314_182348 -n lobe-chat
   kubectl apply -f deploy-postgresql.yaml.backup_20260314_182348 -n lobe-chat
   kubectl apply -f deploy-casdoor.yaml.backup_20260314_182348 -n lobe-chat
   ```

2. 恢复 Secrets（如果需要）：
   ```bash
   kubectl apply -f secrets_backup_20260314_182304.yaml -n lobe-chat
   ```

### 阶段4：启动 v1 服务
```bash
# 启动服务
kubectl scale deployment/lobe --replicas=1 -n lobe-chat
kubectl scale deployment/casdoor --replicas=1 -n lobe-chat
kubectl scale deployment/postgresql --replicas=1 -n lobe-chat

# 检查状态
kubectl get pods -n lobe-chat -w
```

## 回滚风险评估
1. **数据一致性风险**：如果 v2 版本已写入新数据，回滚到 v1 可能导致数据丢失
2. **数据库兼容性风险**：PostgreSQL 17 数据可能不完全兼容 v1 应用
3. **用户中断风险**：回滚过程需要服务重启，造成额外中断

## 回滚决策检查点
执行回滚前确认：
- [ ] 已尝试所有故障修复方案
- [ ] 已评估数据丢失风险
- [ ] 已通知所有用户
- [ ] 有明确的回滚时间窗口
- [ ] 备份文件验证通过

## 监控指标
回滚后监控以下指标24小时：
1. 应用可用性（HTTP 200 响应）
2. 用户登录成功率
3. 数据库连接稳定性
4. 聊天功能正常性
